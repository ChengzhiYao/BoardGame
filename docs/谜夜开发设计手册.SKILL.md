## 0. 一句话定位

**「叫上朋友，两个人就能开局。AI 当主持人，实时给你们编一场只属于今晚的故事。」**

- 一个站点、多种玩法：克苏鲁调查跑团（CoC）、海龟汤、真心话大冒险、剧本杀、血染（社交推理）、午夜猫诅咒（牌局）、龙与地下城（D&D）、讲故事。
- 核心体验：**没有固定脚本**——剧情、线索、NPC、结局都由高额 LLM 实时生成，世界随玩家每一个选择而变。
- 房主（host）付费开房（买断"局数"），被邀请的朋友**永远免费**，无需登录。
- 技术栈：Next.js 14（App Router）+ TypeScript + Tailwind + Supabase（Postgres/Auth/Realtime/Storage）+ Vercel + Stripe + Anthropic/OpenAI/DeepSeek + OpenAI 图片 + Azure TTS。

## 1. MystNight 设计系统（视觉风格）

设计源在 Figma（MystNight Design System），通过 `design/` 落到代码：
- `design/mystnight-preset.cjs` — Tailwind preset，1:1 镜像 Figma 变量（Primitives → Semantic → Scale），在 `tailwind.config.ts` 用 `presets: [mystnight]` 接入。
- `design/tokens.css` — 同一套 token 的 CSS 变量，给运行时主题化用。
- `components/ui/` — 与 Figma 组件对应的起步组件（Button / Badge / Card / Chip / ScoreBar），**只用语义 token 类**（`bg-bg-surface`、`text-text-primary`、`border-border/30`、`bg-accent`、`state-success`…）。
- `components/ui/*.figma.tsx` + `figma.config.json` — Figma Code Connect（`*.figma.tsx` 已在 tsconfig 排除出 Next 构建）。

语义别名（**写代码优先用语义类，不要写死十六进制**）：
`bg.page/surface/raised/hover`、`text.primary/secondary/muted/inverse`、`accent(.hover)`、`mystic(.hover)`、`border`、`state.success/warning/danger/info`。

- 字体：`serif = ['Noto Serif SC','Georgia','serif']`（全站衬线，营造旧书/羊皮纸感）。

- 间距标度：`1=4 2=8 3=12 4=16 5=20 6=24 8=32`（px）。

### 全局样式（`app/globals.css`）
- body：`#0c0d10` 背景 + `#d8c7a8` 文字 + 衬线字体。

### 美术风格锁（出图唯一标准 · `lib/image/style.ts`）
**所有出图都必须经过 `lib/image/style.ts` 的 builder**，保证整局视觉统一。
- 默认 `ART_STYLE` = 阴郁恐怖插画 / 铜版蚀刻质感 / 低饱和暗调（炭黑、灰烬、深青、暗干血红）/ 浓雾重影 / 胶片颗粒 / 克制不卡通；**画面无文字、无水印、无边框**。
- 按题材自动切风格 `artStyleFor(theme)`：中式民俗（水墨枯笔、朱砂墨黑）、日式怪谈（浮世绘、靛蓝苍白）、现代都市（冷调胶片、霓虹惨白）。
- 图片**分类型构图** `ImageType`：`scene_image`（地点空镜，优先）/ `npc_portrait`（半身肖像）/ `clue_evidence`（实物特写）/ `monster_image`（半隐怪物）/ `event_illustration`（关键事件定格）。**禁止把抽象概念/线索标题直接出图**，subject 必须是具体可画的物理画面。
- 专用 builder：`buildAvatarPrompt`（CoC 角色头像，按属性推导外形）、`buildJbsPortraitPrompt`（剧本杀，画风随本剧题材不固定）、`buildDndPortraitPrompt`（奇幻写实，装备贴合职业）。
- 新增任何出图入口 → 一律走这里的 builder，不要在路由里手拼 prompt。

## 2. 整体架构（网站怎么运作）

```
浏览器(匿名/Google 登录) ──RLS受限读 + Realtime订阅──► Supabase(Postgres/Auth/Realtime/Storage)
        │                                                        ▲
        │  fetch /api/*（玩家行动、开房、生成…）                   │ service_role 绕过RLS写
        ▼                                                        │
Next.js Route Handlers（服务端，唯一能碰密钥与"真相"的地方）───────┘
        │  callLLM / callLLMJson（Claude/GPT/DeepSeek 可切换）
        │  generateImage（OpenAI 图片）、Azure TTS、Stripe
        ▼
   写回 Supabase → Realtime 自动广播给两名玩家 → 前端响应式更新
```

### 关键分层原则
1. **前端永远拿不到"真相"**。隐藏案件档案、记忆摘要、骰子随机数等全部在服务端生成与判定，落库到只有 `service_role` 能读的表（见 §3）。前端只播放结果/动画。
2. **三种 Supabase 客户端**（`lib/supabase/`）：
   - `client.ts` `createClient()` — 浏览器端，anon key，受 RLS，用于 Realtime 订阅与读玩家自己有权看的数据。
   - `server.ts` `createServerClient()` — 服务端跟随登录身份，受 RLS。
   - `server.ts` `createAdminClient()` — service_role，**绕过 RLS**，读真相 / 写不可篡改日志 / 改房间状态。**绝不能在前端 import**。
3. **所有"权威写"走 admin 客户端 + 服务端校验**（座位、满员、开房资格、回合串行）。玩家直接经 RLS 能写的只有：自己的角色卡（建卡阶段）、自己的行动消息、自身在线状态。
4. **目录约定**：`app/api/<mode|domain>/<action>/route.ts` = 服务端动作；`app/room/[id]/<Mode>Room.tsx` = 各模式房间 UI；`lib/<mode>/` = 该模式的纯逻辑/提示词；`lib/<横切关注>/`（llm/supabase/billing/image/audio/i18n/kp/review）= 通用能力。

### 前端房间结构（`app/room/[id]/`）
`page.tsx` → `RoomClient.tsx`（订阅 Realtime、分发状态）→ 按 `room.mode` 渲染对应房间组件：
`Dashboard`（CoC 主面板）/ `BotcRoom` / `DndRoom` / `JbsRoom` / `MccRoom` / `SoupRoom` / `TDRoom` / `StoryRoom`，外加 `CharacterFlow`（建卡向导）、`ModuleSelection`（选模组）、`Stepper`（阶段进度）、`AudioManager`（动态配乐）、`useTTS/voiceControl`（朗读）。

### 部署（Vercel）
- `vercel.json` 定时任务：`/api/rooms/sweep` 每天 04:00（清理过期房间，`lib/rooms/sweep.ts`）。
- `next.config.js`：`reactStrictMode`，且 `typescript.ignoreBuildErrors` + `eslint.ignoreDuringBuilds`（先上线，告警后清）。
- 上线流程见 `上线部署指南.md`：GitHub Desktop 推送 → Vercel Import → 在 Vercel 填环境变量（照抄 `.env.local`）→ Deploy → 回填 `NEXT_PUBLIC_SITE_URL`。之后 push 自动重部署。
- **免费版函数最长 60s**：案件生成/结算极少数情况可能偏久；大主调用因此**不重试**（怕超时），靠宽容 JSON 解析兜底（见 §7）。要更稳可升级 Vercel Pro。

### 环境变量（`.env.example`）
Supabase（URL / anon / service_role）；LLM（`LLM_PROVIDER`、`ANTHROPIC_API_KEY`、`OPENAI_API_KEY`、可选 DeepSeek；`LLM_MAIN_MODEL` 强模型、`LLM_AUX_MODEL` 便宜模型、`LLM_FALLBACK_MODEL`）；图片（`IMAGE_PROVIDER`/`IMAGE_API_KEY`/`IMAGE_MODEL`）；`NEXT_PUBLIC_SITE_URL`；TTS（`TTS_PROVIDER=azure`、`AZURE_SPEECH_KEY/REGION`）；Stripe（`STRIPE_SECRET_KEY` 等）。**带 service_role / api_key 的全是机密，只在服务端用，绝不下发前端。**

## 3. 数据库（Supabase）· 安全模型

基础建表在 `supabase/schema.sql`，增量改动统一收在 `supabase/migration_RUN_THIS.sql`（全部"不存在才添加 / 可重复执行"）。**任何 schema 改动都追加到这个总迁移文件**，方便上线前一次跑完。

### 三条安全红线（务必延续）
1. **真相保密**：`hidden_case_files`（truth/mastermind/supernatural/key_clues/timeline_true/endings…）**不创建任何 RLS select 策略** → anon/authenticated 一律被拒，只有 service_role 能读写。`memory_summaries`、`api_usage`、`error_logs` 同理（仅服务端/管理员）。
2. **防篡改**：`dice_rolls` 与 `san_logs` 用 `block_modify()` 触发器禁止任何 update/delete（连 service_role 也挡）；只允许服务端 insert。`hidden_case_files` 锁定后（`locked_at` 非空）核心字段不可改（`lock_case_file()` 触发器）。
3. **私人线索物理隔离**：`clues/npcs/timeline_events` 带 `visible_to ∈ {all,A,B}`，RLS 用 `my_seat(room_id)` 只下发 `all` 或本座位的行；给前端的 `clues_player` 视图用 `security_invoker=true` 裁掉真相元标记（`is_key/is_red_herring`）且继承查询者 RLS。

### 核心表速查
- 身份/房间：`users`（auth 触发自动建 profile）、`rooms`（状态机字段：`status` + `game_state`；`mode`、`language`、`turn_count`、`image_budget/used`、`suspicion`、世界时钟/记忆/回合制等都挂在 rooms）、`players`（座位 A/B、最多 2）、`room_settings`。
- CoC 内容：`characters`（CoC 7e 角色卡：八属性、HP/SAN/Luck/MOV/DB/Build、`skills jsonb`、`status_flags`、avatar/inventory 等）、`messages`（共享剧情流，`visibility=public/player_a/player_b`）、`campaigns`（玩家可见模组层，`status=draft/locked`）、`hidden_case_files`（机密真相）、`clues/npcs/locations/timeline_events`、`dice_rolls/san_logs`（不可篡改）、`images`（生成图）、`memory_summaries`（AI 记忆，含 `vector(1536)` pgvector）。
- 计费/后台：`profiles`（credits、is_whitelisted、free_granted）、`whitelist_emails`、`api_usage`（成本统计）、`feedback`、`error_logs`。
- 其它模式（botc/dnd/mcc/jbs/soup/td/story）多把状态塞进 `rooms` 的 jsonb 列或自有迁移表（见各 `lib/<mode>/db.ts`）。
- **Realtime publication**：需要实时同步的表（rooms/players/characters/messages/campaigns/clues/npcs/locations/timeline_events/dice_rolls/san_logs/images）已加入 `supabase_realtime`。新表若要前端实时刷新，记得 `alter publication supabase_realtime add table`。

## 4. 双模型 LLM 适配器（`lib/llm/index.ts`）

上层只调两个函数，不关心底层是哪家：
- `callLLM(req)` — 通用文本。
- `callLLMJson<T>(req)` — **强制结构化 JSON 输出**（KP 回合、模组生成等都用它），自带：尾部 JSON 守卫指令、宽容解析（去 ```fence```、截取首 `{` 到末 `}`）、失败兜底（主 provider 拿不到合法 JSON → 退回 OpenAI `LLM_FALLBACK_MODEL` 再试一次）。

约定：
- `tier: 'main' | 'aux'`。**main = 强模型**（真相生成、回合主持、写故事），**aux = 便宜模型**（记忆摘要、行动解析、配图 prompt、分类）——分层省钱是硬约定。
- `provider: 'anthropic' | 'openai' | 'deepseek'`，默认读 `LLM_PROVIDER`；DeepSeek 复用 OpenAI SDK（换 baseURL + key）。
- 温度：真相相关用低温（默认 JSON 0.4、叙述 0.6）。
- **重试策略**：默认仅 `aux` 重试一次；`main` 大调用不重试（避免 Vercel 60s 超时），只靠宽容解析 + OpenAI 兜底。
- **每次调用后写 `api_usage`**（kind=llm_main/llm_aux/image、model、tokens、latency），用于成本统计与熔断。
- **只在服务端 import**。

## 5. 各游戏模式总览 + 统一开发范式

| mode | 名称 | 核心逻辑位置 | 玩法骨架 |
|---|---|---|---|
| `coc` | 克苏鲁调查跑团 | `lib/coc/` + `lib/kp/` | 选模组→锁真相→双人回合制调查，骰子/SAN/线索/嫌疑/世界时钟 |
| `soup` | 海龟汤 | `lib/soup/` | 生成汤面+汤底→玩家问是非题→AI 裁定→揭晓判定 |
| `td` | 真心话大冒险 | `lib/td/` | 内置题库（零 token）+ AI 按尺度/环境生成，严格安全红线 |
| `jbs` | 剧本杀 | `lib/jbs/` | 生成剧本→分发角色→分幕推进→搜证→投票→结算，画风随本剧 |
| `botc` | 血染（社交推理） | `lib/botc/` | 原创身份体系→逐角色叫醒夜结算→白天 AI 推理发言→投票处决 |
| `mcc` | 午夜猫诅咒（牌局） | `lib/mcc/` | 原创"赌运气"猫主题牌局，纯函数确定性引擎 + AI 出牌 bot |
| `dnd` | 龙与地下城 | `lib/dnd/` | 建卡→选冒险→锁隐藏蓝图→探索/检定/战斗（引擎确定性）/休整 |
| `story` | 讲故事 | `lib/story/` | 按参数生成 3 个方案→写作蓝图→全文→多维评分→TTS 朗读 |

### 反复出现的统一范式（设计这套东西的"内功心法"）
几乎每个 AI 模式都遵循同一套结构，新模式照抄即可：

1. **生成"玩家可见层" vs 隐藏"真相/蓝图"两套数据**。先生成给玩家挑的选项（模组/冒险/剧本，**不含真相/凶手/结局**）；玩家选定后，后台再生成完整隐藏档案并**锁定**到 service_role-only 表。例：`buildModuleGenPrompt`→`buildCaseLockPrompt`（CoC）、`buildDndQuestsPrompt`→`buildDndBlueprintPrompt`（D&D）。
2. **质量门（review/score）**。隐藏档案生成后用一个挑剔的审稿 prompt 打分，低于阈值就重生成。`lib/kp/review.ts`（`QUALITY_PASS=72`）、`lib/dnd`（`DND_QUALITY_PASS=70`）、`lib/review/quality.ts`。
3. **两段式回合（解析器 → 叙述器）**。先用便宜模型把玩家自由文本**解析**成判定计划（意图/澄清/骰检/SAN/敌意回击），**绝不写剧情**；服务端按计划掷骰/扣血/扣 SAN（随机数与判定在服务端、落不可篡改日志）；再用强模型**据结果叙述后果**并输出结构化 JSON（叙事+引导+线索+状态变更+配图+音效+结局）。见 `lib/kp/resolver.ts` + `lib/kp/prompt.ts`，编排在 `app/api/turn/route.ts`。
4. **机制确定性、叙事 AI**。骰子（`lib/coc/dice.ts` CoC 7e 成功等级/奖惩骰/对抗）、牌局（`lib/mcc/engine.ts` 纯函数）、D&D 战斗（`lib/dnd/engine.ts`）都由**确定性代码**算，AI 只负责叙事和生成数值化的敌人/怪物。这样省钱、可控、可复现。
5. **滚动记忆压缩**（长局不爆 token）。`lib/kp/memory.ts`：每 `SUMMARIZE_EVERY=4` 回合用 aux 模型把历史压成 `summary`(≤300字) + `key_facts[]`，注入后续 system prompt。
6. **结构化 JSON 契约**。回合输出是一个大 JSON（narration/guidance/clue_updates/npc_updates/state_changes/image_suggestion/scene_state/sfx/progress…），路由按字段落库并广播。新模式定义自己的 JSON 契约时，保持"不需要的数组留空 []"的约定。

### KP（守秘人）提示词的设计准则（`lib/kp/prompt.ts`，是全项目最精华的部分）
延续这些"灵魂"约定，别写成温吞解谜：
- **绝对禁止说教**：只描述行为本身、判定结果、世界如何反应；不替玩家反思、不劝阻。
- **恐怖渲染要真吓人**：先酝酿再爆发、文字版 jump scare、感官要生理具体、NPC"差一点点不对"、别把怪物讲透、对低理智玩家投放**私有幻觉**（`hallucinations`，只发给该座位）。
- **世界自己在走**：世界时钟（`world_clock`）随回合到点触发后果；NPC 有 wants/记忆/offscreen 行动；嫌疑值（suspicion）阈值驱动巡逻/封锁/警察介入/追捕。
- **玩家有权走"反派/越界路线"**：召唤/放纵异常、与凶手同流、献祭、自己成为怪物都是合法可达成的路线，要**给他实现的手段**，绝不用"你做不到"堵死。
- **结局由"路线与态度"驱动、允许涌现**：按精神匹配 `ending_conditions/hidden_endings`，玩家用预设外但逻辑自洽的方式收束也可即兴承认。
- **多线索拒绝重复**：案件分 5 条线（A 建筑史/B 失踪死亡/C NPC 异常/D 超自然/E 关键物品仪式），每条新线索必须含新信息，查无可查就明说并引向未探索方向。
- **题材一致性**：严格贴合所选题材（中式/日式/克苏鲁/校园…）的意象与逻辑，骰子/SAN 是通用机制但叙事风味随题材走。

## 6. 新增一个游戏模式 · 标准步骤（Checklist）

以加一个新模式 `xyz` 为例，按既有约定逐步做：

1. **登记 mode**：在 `app/api/rooms/route.ts` 的允许列表加 `'xyz'`；在 `lib/i18n.ts` 的 `ZH/EN` 字典加 `mode_xyz` / `create_xyz` 等文案（**中英都要**）。落地页 `app/page.tsx` 与 `components/GameShowcase.tsx` 加入口。
2. **数据**：状态优先塞进 `rooms` 的 jsonb 列（少建表）；确需新表则写进 `supabase/migration_RUN_THIS.sql`（`add column/table if not exists`），并按 §3 三红线设好 RLS（真相 service_role-only、日志不可篡改、私人信息按座位隔离）；要前端实时刷新就加进 realtime publication。
3. **逻辑层 `lib/xyz/`**：
   - `prompt.ts` — 所有 AI 提示词构建函数（生成选项 / 锁定隐藏档案 / 回合裁定 / 审查评分），遵循"玩家可见层 vs 隐藏层 + 质量门 + 结构化 JSON"范式。
   - 确定性机制写成**纯函数引擎**（如 `engine.ts`），随机数与判定放服务端。
   - 需要 `db.ts` 就放该模式的取数/落库辅助。
4. **API 路由 `app/api/xyz/<action>/route.ts`**：`createServerClient` 取登录用户 → `createAdminClient` 校验成员/座位/阶段 → `callLLMJson`（main 叙事、aux 解析）→ 服务端判定 → 落库（含 `api_usage`、必要时 `error_logs`）→ 返回。**密钥与真相只在这里碰**。
5. **前端 `app/room/[id]/XyzRoom.tsx`**：在 `RoomClient.tsx` 按 `room.mode` 分发；只用 §1 的语义 token 类与 `components/ui`；订阅 Realtime 响应式更新；动画加在 `globals.css`（`xyz` 前缀）。
6. **出图/音效/朗读**（按需）：出图一律走 `lib/image/style.ts` builder；动态配乐进 `lib/audio/`（`manifest.ts` + `<mode>Cue.ts`，音频文件放 `public/audio/<mode>/`）；朗读用 `app/api/*/tts` + Azure。
7. **计费**：开房统一走 `getEntitlement`（白名单/局数门）+ 成功扣 1 局，局数对所有模式通用，无需单独计费逻辑。
8. **成本控制**：贵活用 main、轻活用 aux；记得写 `api_usage`；出图走 `image_budget/used` 预算；长局接 §5.5 记忆压缩。

## 7. 健壮性与省钱的既有约定（别破坏）

- **JSON 解析容错**：`callLLMJson` 已处理 fence/思考前缀/空响应；新提示词务必在结尾要求"只输出一个合法 JSON 对象"。
- **Vercel 60s**：main 大调用不重试；把"解析（aux，可重试）"和"叙述（main，不重试）"拆开，避免一次调用又大又会重试。
- **熔断/预算**：图片按 `image_budget` 限额；`api_usage` 可做成本统计与封顶；公开网址会真实扣 host 的 API 额度，故"别到处公开发链接 + 在 OpenAI 后台设上限"。
- **匿名登录**：被邀请玩家走 Supabase Anonymous sign-in（须在 Supabase 开启），免注册即得身份；host 用 Google 登录以记住付费状态。
- **回合串行化**：`rooms.current_lock` + advisory lock 防并发回合；双人回合制用 `pending_actions/player_a_ready/player_b_ready/resolution_status` 协调"两人都提交才结算"。

## 8. 国际化（`lib/i18n.ts`）

- UI 文案：`ZH/EN` 两本字典 + `tr(lang)(key, vars)`；**新增文案中英都要补**。
- AI 输出语言：`langDirective(lang)` 拼到 system prompt 末尾控制面向玩家的输出语言（JSON 的 key 保持不变，只翻 value）。
- 语言来源：开房前用 cookie/localStorage（`getClientLang/setClientLang`，中文浏览器默认 zh，其余 en）；游戏内用 `room.language`（两名玩家一致）。
